## 1. Phase-1-Kandidaten — @bpdjs/core

### 1.1 Logger
**Keine dedizierte Logger-Instanz vorhanden.** Logging läuft direkt über `console`, kein Wrapper, keine LogLevels.

- ~70 `try { } catch { }`-Blöcke in `main.mjs`, alle nutzen `toErrorMessage()` (`main.mjs:99–104`)
- Standard-Pattern: `throw new Error(toErrorMessage(error))` (z. B. `main.mjs:314, 330`)

**Empfehlung:** Logger ist *Nice-to-Have*. Phase 1 könnte `toErrorMessage` + minimalen `centralLog()`-Helper extrahieren — oder es bleibt vorerst bei `console`, und Logger kommt später, wenn Bedarf entsteht.

### 1.2 Config
- **Quelle:** `resolveConfig()` aus `scanner/cdp-scanner.mjs` (Import in `main.mjs:24`)
- **Defaults:** `DEFAULTS`-Objekt aus `scanner/cdp-scanner.mjs` (Import `:26`)
- **App-Paths:** `app.getPath('userData')` (`:78, :210`), `home` (`:448`), `temp` (`:487`)
- **Runtime-Merge:** `buildRuntimeConfig()` (`:282–296`) mergt `rawConfig` + AppInfo
- **Config-Keys:** `canonicalAppPath`, `beatportSessionPartition`, `userDataPath`, `execPath`, `appPath`, `version`, `buildId`, `appVersion`, `runtimeClientFactory`

**Empfehlung Phase 1:** `resolveConfig` + `DEFAULTS` + `buildRuntimeConfig` nach `@bpdjs/core/config` — nur Pfade + Umgebung, **Scanner-Logik bleibt in cdp-scanner.mjs**.

### 1.3 EventBus
**Keine EventBus-Instanz.** Event-Kommunikation geht ausschließlich über IPC + Electron-App-Events:
- `ipcMain.send/on` für Progress-Events (z. B. `sync:batch-progress` `:1838, :1847`)
- `app.on("activate"|"window-all-closed"|"before-quit")` (`:1931, :1938, :1944`)
- `BrowserWindow.on("closed")` (`:1660`)

**Bewertung:** Interne Module (scanner → cache → auth) nutzen Callback-Ketten statt Events. Ein EventBus wäre **neu eingeführt**, kein Refactor. Entscheidung: vorerst weglassen oder als separater Mini-Step behandeln.

### 2.3 Wiederholungspatterns

**Der Goldfund — `ipcHandle`-Wrapper (`main.mjs:307–316`):**
```js
ipcHandle(channel, fn) {
  ipcMain.handle(channel, async (_event, rawConfig, ...args) => {
    try {
      const config = await buildRuntimeConfig(rawConfig || {});
      return await fn(config, ...args);
    } catch (error) {
      throw new Error(toErrorMessage(error));
    }
  });
}
```

- **34 Handler** nutzen diesen Wrapper → ideal für `@bpdjs/ipc-router`
- **61 Handler** sind direkt `ipcMain.handle` (kein Config-Inject)
- Progress-Streaming via `event.sender.send(...)` (`:1838, :1847, :1868`) — kein generisches Pattern, perfekt für `.sendProgress()`-Helper

## 3. Synergie-Punkte (Phase 1 ↔ Phase 2)

| Punkt | Konkret |
|-------|---------|
| **Logger in IPC** | 70+ `catch { throw new Error(toErrorMessage(error)) }` — zentraler Logger würde alle Handler logbar machen |
| **Config in IPC** | `ipcHandle`-Wrapper macht `buildRuntimeConfig(rawConfig)`-Inject → Router muss diesen Pattern verstehen |
| **Progress-Streaming** | `event.sender.send("sync:batch-progress", ...)` → `router.sendProgress(channel, payload)` als Abstraktion |
| **EventBus** | Optional: scanner→cache→auth Callback-Ketten könnten Events werden (aber neu, nicht im Bestand) |
