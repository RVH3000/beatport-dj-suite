# v4.3 Build-Pipeline-Plan — @bpdjs/* Workspace-Symlink-Problem

**Stand:** 2026-05-13
**Branch:** feat/v4.2 (Tag v4.2.17)
**Erstellt von:** feature-dev:code-architect Agent-Lauf
**Kontext:** Vorbereitung für M5-Refactor (98 IPC-Handler → ~30 @bpdjs/-Imports)

---

## 1. Lage — Was heute funktioniert, was nicht

### Heute funktioniert

- `npm run desktop:dev` — Dev-Start via `electron .` mit Workspace-Symlinks: funktioniert,
  weil Node.js Symlinks zu Verzeichnissen auflöst.
- `npm test` — 905/905 grün; Tests laufen direkt im Workspace ohne Build-Step.
- `npm run desktop:dist:mac` — baut eine funktionierende `.app` via `scripts/build-mac.sh`.
  Das Skript dereferenziert die 13 Symlinks unter `node_modules/@bpdjs/` vor dem Build
  (echte Verzeichniskopien per `cp -RL`) und stellt sie per `trap`-Handler im EXIT danach
  wieder her. `package.json` `files`-Array enthält `"node_modules/@bpdjs/**/*"` — das
  greift genau dann, wenn echte Verzeichnisse statt Symlinks vorliegen.
- `main.mjs` (v4.2.17) hat aktuell nur einen relativen Import:
  `from "../packages/core/index.js"` (Zeile 14). Ein `@bpdjs/`-Import-Name kommt noch
  nicht vor.

### Heute nicht skalierbar

- M5 bringt ~30 statische `import { … } from "@bpdjs/X"` in main.mjs.
  Der Symlink-Deref-Workaround in `build-mac.sh` funktioniert auch dann — aber:
  - Der `trap`-basierte Restore ist fragil: bricht npm nach dem Build ab oder
    läuft ein zweiter Build-Prozess parallel, landen echte Kopien dauerhaft in
    `node_modules/@bpdjs/` und werden in Git als untracked angezeigt.
  - `cp -RL` kopiert bei M5 13 Pakete mit mehreren hundert Dateien — Build-Zeit
    steigt unkontrolliert.
  - Das Muster ist unsichtbar: ein neuer Contributor sieht `build-mac.sh`, versteht
    nicht warum echte Kopien temporär notwendig sind, und baut mit dem falschen
    Kommando (`desktop:dist:mac:raw` oder `desktop:dir:mac`), was sofort bricht.
- `engine-db-core` ist ein TypeScript-Paket (`tsc`-Build → `dist/`). Sein `main`
  zeigt auf `./dist/index.js`. Beim Deref-Workaround wird `dist/` nur dann mitkopiert,
  wenn `tsc` vorher lief — das ist im Skript nicht gesichert.

### Nicht betroffen (stabil bleiben)

- Relative Imports innerhalb von `electron-app/` (`./integrations/…`, `./api/…`)
- Dynamic imports auf `node:fs`, `electron`, `groof-client.mjs` — alle umgehen
  den `@bpdjs/`-Pfad vollständig.
- Tests — laufen nie durch electron-builder, nutzen Workspace-Symlinks direkt.

## 2. Drei Strategien

### Strategie 1 — esbuild als Pre-Build-Bundler

**Idee:** Vor `electron-builder` läuft esbuild. Es nimmt `electron-app/main.mjs`
als Einstieg, bündelt alle `@bpdjs/*`-Imports und gibt eine einzelne
`electron-app/main.bundle.mjs` aus. electron-builder packt das Bundle.

**Externals:** `electron`, `node:*`, `better-sqlite3` (native), ggf. weitere
native modules. Alles andere wird eingebettet.

**`engine-db-core`-Sonderfall:** Das TS-Paket muss vor esbuild gebaut sein
(`dist/` existiert). esbuild liest `./dist/index.js` via den `exports`-Eintrag —
TypeScript-Quellen werden nicht berührt.

**Dynamic Imports:** esbuild kann dynamische Imports auf relative Pfade
(`./integrations/m3u-exporter.mjs`, `./api/groof-client.mjs` usw.) als
Externals erhalten oder separat in das Output-Dir kopiert werden. Diese sind
alle relativ und bleiben nach dem Bundle unverändert.

| Kriterium | Bewertung |
|---|---|
| Einführungsaufwand | 3–5 Stunden (esbuild als devDep, Config, externals-Liste, build-mac.sh ersetzen) |
| Risiko bei Einführung | Mittel — externals-Liste muss vollständig sein, sonst fehlt etwas im Bundle |
| Wartungsaufwand/Monat | Gering — neue `@bpdjs/`-Pakete werden automatisch gebündelt, keine Config-Änderung |
| Auswirkung auf Tests | Keine — Tests laufen weiterhin mit Node.js direkt, kein esbuild |
| In 2 Jahren | Sehr gut — 30+ Pakete sind kein Problem, Bundle ist eine Datei, Debug via Source-Maps |
| Stolperfallen | Dynamic imports, native modules, engine-db-core muss vor esbuild gebaut sein |

**Urteil:** Technisch sauberste Lösung, aber höchster Einführungsaufwand.
Externe Abhängigkeitsliste (externals) muss gepflegt werden.

---

### Strategie 2 — `file:`-Dependencies statt Workspaces

**Idee:** Alle `@bpdjs/*` in der Root-`package.json` aus den Workspaces entfernen
und als `"file:packages/X"` in `dependencies` eintragen. `npm install` erzeugt
dann echte Kopien (keine Symlinks) unter `node_modules/@bpdjs/`. electron-builder
packt sie als Production-Deps — `build-mac.sh` wird überflüssig.

**Development-Konsequenz:** `npm install` nach jeder Änderung an einem Paket
nötig, oder `npm run build --workspace=@bpdjs/X`. Workspace-Features (hoisting,
cross-package `*`-dependencies) funktionieren nicht mehr.

| Kriterium | Bewertung |
|---|---|
| Einführungsaufwand | 1–2 Stunden (package.json umschreiben, npm install, smoke test) |
| Risiko bei Einführung | Hoch — `*`-Dependencies zwischen Paketen (`@bpdjs/engine-db` → `@bpdjs/core`) werden zu echten Versions-Constraints, können zu doppelten Instanzen führen |
| Wartungsaufwand/Monat | Hoch — jede Paket-Änderung erfordert `npm install` oder explizites `file:`-Update |
| Auswirkung auf Tests | Mittel — Testlauf liest aus `node_modules/` statt Workspace-Symlinks; Gefahr von veralteten Kopien |
| In 2 Jahren | Schlecht — 14 Pakete mit gegenseitigen Deps als `file:` ist unhandhabbar, keine tree-shaking-Unterstützung |
| Stolperfallen | Doppelte Instanzen bei intra-Paket-Dependencies, Development-Workflow bricht |

**Urteil:** Geringster Einführungsaufwand, aber zerstört den Development-Workflow
und skaliert nicht auf das M5-Ökosystem. Nicht empfohlen.

---

### Strategie 3 — Hybrid: Workspaces Development + Production-Manifest vor Build

**Idee:** Workspace-Setup bleibt unverändert für Development und Tests.
Vor dem Build schreibt ein Node.js-Skript eine temporäre `package.json`-Kopie
mit `@bpdjs/*` als `file:`-Einträge, führt `npm install --production` in einem
Temp-Verzeichnis aus und kopiert das Ergebnis nach `node_modules/@bpdjs/`.
Nach dem Build wird der Original-Zustand wiederhergestellt.

Im Prinzip ist dies eine Verfeinerung des aktuellen `build-mac.sh`-Workarounds:
statt `cp -RL` wird `npm install` verwendet, was korrekte intra-Paket-Auflösung
garantiert.

| Kriterium | Bewertung |
|---|---|
| Einführungsaufwand | 2–4 Stunden (Skript, Fehlerbehandlung, Tests) |
| Risiko bei Einführung | Mittel — `npm install` in Temp-Dir mit file: kann bei Netz-Problemen hängen (prefer-offline mitgeben) |
| Wartungsaufwand/Monat | Mittel — Skript muss gepflegt werden wenn neue Pakete hinzukommen |
| Auswirkung auf Tests | Keine — Workspace bleibt intakt |
| In 2 Jahren | Mittel — Ansatz ist konzeptuell sauber, aber das Skript bleibt ein versteckter Build-Schritt |
| Stolperfallen | Race conditions bei parallelen Builds, engine-db-core braucht tsc-Build vor npm install |

**Urteil:** Guter Mittelweg wenn esbuild zu viel Aufwand ist. Sauberer als
Strategie 1 nicht, aber sicherer als der aktuelle `cp -RL`-Workaround.

## 3. Empfehlung

**Strategie 1 — esbuild als Pre-Build-Bundler** ist die richtige Wahl für v4.3.

Begründung:

1. **M5 ist der Wendepunkt.** Mit ~30 statischen `@bpdjs/`-Imports in main.mjs
   ist der Symlink-Deref-Workaround kein temporärer Hack mehr, sondern Infrastruktur.
   Infrastruktur soll explizit und wartbar sein — ein esbuild-Config ist das.

2. **Build-Zeit ist kontrollierbar.** esbuild bündelt 14 Pakete in unter 1 Sekunde.
   Der `cp -RL`-Workaround skaliert linear mit Dateizahl, esbuild nicht.

3. **engine-db-core passt.** `dist/` ist bereits gebaut; esbuild liest fertige JS-Dateien.
   Kein TypeScript-Verständnis nötig.

4. **Tests bleiben vollständig unberührt.** Das ist die härteste Anforderung — esbuild
   läuft nur im Build-Pfad, nie im Test-Pfad.

5. **Langlebig.** Wenn die Suite in 2 Jahren 30+ Pakete hat, ist die esbuild-Config
   unverändert. Source-Maps erlauben Debugging des Bundles im gepackten .app.

**Trade-offs die akzeptiert werden:**

- Externals-Liste muss gepflegt werden (Grundregel: alles was nicht `@bpdjs/` ist
  und nicht pure JS ist, kommt auf die Liste).
- Dynamic imports auf relative Pfade brauchen explizite Behandlung (als external
  markieren oder in das Output-Dir kopieren).
- Build ist zweistufig: `tsc` (engine-db-core) → esbuild → electron-builder.

## 4. Konkreter Implementierungs-Plan

### Schritt 1 — Vorbereitung und Abhängigkeiten

- `esbuild` als devDependency in Root-`package.json` hinzufügen (aktuelle stable, z.B. `^0.25`).
- `esbuild.config.mjs` im Repo-Root anlegen (Datei existiert noch nicht).
  Inhalt-Stichworte (kein Code hier): Entrypoint `electron-app/main.mjs`,
  outfile `electron-app/main.bundle.mjs`, format `esm`, platform `node`,
  bundle `true`, sourcemap `true`.
  Externals: `electron`, alle `node:*`-Protokoll-Imports, `better-sqlite3`,
  alle relativen Dynamic-Import-Pfade (`./integrations/*`, `./api/*`).
- Neue npm-Scripts in Root-`package.json`:
  - `"build:main"` — ruft `esbuild.config.mjs` auf
  - `"build:ts-packages"` — ruft `npm run build --workspace=@bpdjs/engine-db-core` auf
  - `"prebuild"` — Sequenz: `build:ts-packages` → `build:main`

### Schritt 2 — electron-builder auf Bundle umstellen

- In `package.json` `build.main` auf `"electron-app/main.bundle.mjs"` setzen
  (aktuell implizit über Root-`"main": "electron-app/main.mjs"`).
- `build.files`-Array bereinigen:
  - `"node_modules/@bpdjs/**/*"` entfernen (wird nicht mehr benötigt, da gebündelt).
  - `"electron-app/main.mjs"` durch `"electron-app/main.bundle.mjs"` ersetzen.
  - `"electron-app/integrations/**/*"` und `"electron-app/api/**/*"` explizit
    aufführen (Dynamic-Import-Targets, die esbuild als external behandelt).
  - `"packages/**/*"` aus `files` entfernen (Source wird nicht mehr gebraucht,
    nur `dist/` von engine-db-core für potenzielle Runtime-Referenzen).
  - `"packages/engine-db-core/dist/**/*"` behalten falls direkte Pfad-Referenzen
    außerhalb des Bundles existieren.
- `asarUnpack` bleibt unverändert.

### Schritt 3 — build-mac.sh ersetzen

- `scripts/build-mac.sh` wird auf folgendes Gerüst reduziert:
  1. `npm run prebuild` (tsc + esbuild)
  2. `CSC_IDENTITY_AUTO_DISCOVERY=false electron-builder --mac dmg zip`
- Der gesamte Symlink-Deref/Restore-Block (Zeilen 14–29) entfällt.
- Das `trap`-Konstrukt entfällt.
- `desktop:dist:mac:raw` in package.json bleibt als direktes electron-builder-Kommando
  erhalten, wird aber nur nach manuellem `npm run prebuild` genutzt.

### Schritt 4 — Dev-Modus sichern

- `desktop:dev` läuft weiterhin mit `electron .` — Einstieg via Root-`"main": "electron-app/main.mjs"`.
  Im Dev-Modus werden Workspace-Symlinks direkt aufgelöst. Das Bundle wird nicht verwendet.
- Der Hot-Reload-Block in main.mjs (Zeile 21–29) bleibt unverändert und greift
  weiterhin nur wenn `!app.isPackaged`.
- `.gitignore` um `electron-app/main.bundle.mjs` und `electron-app/main.bundle.mjs.map` ergänzen.

### Schritt 5 — main.mjs auf @bpdjs/-Imports umstellen (M5-Vorbereitung)

- Erst nach validiertem Build-Schritt (Schritte 1–4 grün) beginnen.
- Bestehenden relativen Import (Zeile 14: `from "../packages/core/index.js"`)
  auf `from "@bpdjs/core"` umschreiben.
- Smoke-Test: Bundle neu bauen, `.app` im `dir`-Modus starten, prüfen ob
  `@bpdjs/core` korrekt aufgelöst wird.
- Danach M5-Imports paketweise hinzufügen (Reihenfolge wie in M5-Mapping Phase A–E).

### Schritt 6 — Externe Dynamic-Import-Targets absichern

- `groof-client.mjs`, `m3u-exporter.mjs`, `engine-analyze-matcher.mjs` sind
  Dynamic-Import-Ziele im Build. Sie müssen ins asar-Bundle oder in `asarUnpack`.
  Vorgehen: in `build.files` explizit aufführen, nicht durch esbuild bündeln.
- Alle `import(path.join(...))` mit variablen Pfaden sind Kandidaten für
  `asarUnpack` — statische `asarUnpack`-Globs können sie abdecken.

## 5. Tests parallel grün halten

**Grundregel:** esbuild berührt nie den Test-Pfad.

- `npm test` ruft `node --test` direkt auf Source-Dateien auf — keine Änderung.
- `npm run test:packages` läuft in Workspace-Paketen — keine Änderung.
- Das Bundle `main.bundle.mjs` ist in `.gitignore` und taucht in keinem
  Test-Glob (`tests/**/*.test.mjs`, `packages/*/tests/**/*.test.mjs`) auf.
- `engine-db-core`-Build (`tsc`) ist ein separater Script-Step (`build:ts-packages`).
  Er läuft nicht automatisch beim Test — `dist/` muss vor Test-Run existieren,
  was heute schon Voraussetzung ist.
- Nach Schritt 5 (M5-Imports): `import "@bpdjs/core"` in main.mjs wird im
  Test-Kontext über den Workspace-Symlink aufgelöst. Kein Unterschied zu heute.
- `tests/app-bundle-smoke.test.mjs` muss nach Schritt 2 auf `main.bundle.mjs`
  als Einstiegspunkt angepasst werden falls er den Bundle-Pfad prüft.

## 7. Stolperfallen — Klar markiert

### engine-db-core muss vor esbuild gebaut sein

`engine-db-core` hat `"main": "./dist/index.js"`. Wenn `dist/` nicht existiert,
schlägt esbuild mit "Cannot resolve @bpdjs/engine-db-core" fehl. Der `prebuild`-Step
in package.json muss `build:ts-packages` vor `build:main` ausführen — Reihenfolge
ist zwingend, nicht optional.

Wenn ein Contributor `npm run build:main` direkt aufruft ohne `prebuild`, bricht
der Build. Lösung: esbuild-Config prüft ob `packages/engine-db-core/dist/index.js`
existiert und gibt einen klaren Fehler aus, statt einen kryptischen Resolve-Fehler.

### Dynamic Imports mit variablen Pfaden

`import(path.join(__dirname, '...'))` mit Laufzeit-Variablen kann esbuild nicht
statisch auflösen. Kandidaten aus main.mjs:

- Zeile 867: `await import("./api/groof-client.mjs")` — statischer String, als
  external markieren und Datei in `build.files` aufführen.
- Zeile 1319: `await import(...)` mit `findFuzzyDuplicates` — prüfen ob Pfad
  statisch ist. Wenn dynamisch: in `asarUnpack` aufführen.
- Zeile 1598: `await import("./integrations/engine-analyze-matcher.mjs")` —
  statischer String, als external markieren.

Faustregel: Alle `./`-Dynamic-Imports in `electron-app/` als esbuild-externals
eintragen und im `build.files`-Array aufführen.

### Native Modules

`better-sqlite3` ist native (`.node`-Datei). esbuild darf es niemals ins Bundle
einbetten — es muss zwingend als external deklariert sein. electron-builder
behandelt es via `asarUnpack` (falls noch nicht vorhanden, hinzufügen).

Wenn neue native modules hinzukommen (z.B. `@napi-rs/...`), müssen sie sofort
auf die externals-Liste. Kein automatischer Schutz.

### `app.isPackaged`-Check für Hot-Reload

Der Hot-Reload-Block (main.mjs Zeile 21–29) nutzt `electron-reload` via
Dynamic Import. Im Bundle läuft er bei `!app.isPackaged` — das ist korrekt,
weil `desktop:dev` das ungebündelte `main.mjs` startet. Im gebauten .app wird
`app.isPackaged === true`, der Block wird nicht ausgeführt. Kein Risiko.

Aber: wenn esbuild das Bundle erstellt und `app.isPackaged` statisch als
`false` auswertet (Dead-Code-Elimination), würde der Hot-Reload-Block ins
Bundle eingebettet. `app.isPackaged` ist ein Runtime-Wert — esbuild kann ihn
nicht statisch auflösen. Kein Problem, aber verifizieren dass der
`electron-reload`-Import als external markiert ist.

### MainWindow-Bug (aus M5-Mapping bekannt)

`mainWindow` ist in main.mjs nicht als `let`-Variable deklariert; `createWindow()`
speichert das BrowserWindow nicht (M5-Mapping Abschnitt 1). Das ist ein bestehender
Bug, unabhängig von der Build-Pipeline. Muss vor oder während M5 Phase A gefixt
werden — nicht erst am Ende. Der esbuild-Plan ist davon unabhängig.

**Update 2026-05-13:** Der mainWindow-Bug wurde bereits in v4.2.13 (Commit `571ae44`)
behoben. `let mainWindow = null` ist deklariert, `createWindow()` setzt es,
`closed`-Event räumt es auf.

### Parallelisierbarkeit

`build:ts-packages` und `build:main` dürfen nicht parallel laufen — esbuild
braucht `dist/` fertig. Kein `&`-Operator im prebuild-Script. Sequenziell.
